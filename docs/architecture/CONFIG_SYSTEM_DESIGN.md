# 配置驱动提单系统设计

最后更新：2026-04-10

## 1. 设计目标

配置系统的目标不是只做 CRUD，而是稳定驱动提单默认值、申请单快照和结果回读。

当前设计目标：

1. 页面默认值来自配置，而不是硬编码
2. 项目上下文、用户权限上下文、申请单实例状态分层清晰
3. 历史订单快照不受后续配置变更污染

## 2. 当前分层

### 2.1 用户权限上下文

由用户权限决定，但不属于项目配置：

- 可见项目列表
- 资源池列表
- 默认资源池
- 提单限制

### 2.2 项目上下文

与项目直接相关：

- 项目阶段
- 参与人候选
- 文件上下文

### 2.3 默认提单模板

当前不引入独立模板表，继续由默认工况配置展开得到：

- 默认参数组
- 默认输出组
- 默认求解器

### 2.4 申请单实例状态

真正提交时形成：

- `conditions`
- `conditionId`
- 各工况参数 / 输出 / 求解器配置
- 参与人选择结果
- 备注
- 关注器件

## 3. 配置驱动规则

### 3.1 默认工况驱动

默认快速提单链路以默认工况配置展开为真源。

这意味着：

- 不是项目直接决定仿真类型
- 不是项目直接决定 `conditionId`
- 而是默认工况决定后续默认参数链路

### 3.2 参数组与输出组过滤

过滤规则固定为：

- 若组合显式关联当前项目，则仅该项目可用
- 若组合未关联任何项目，则视为全部项目可用

### 3.3 项目阶段

项目阶段不是静态字典直接全量下发，而是：

1. 从 `pp_phase` 中取当前项目关联阶段
2. 关联 `phase` 字典取名称
3. 最新 `pp_phase_id` 对应的 `phase_id` 作为默认阶段

## 4. 为什么不能再走旧设计

旧设计的问题在于把以下内容混在了一起：

- 项目
- 仿真类型
- 初始化接口
- 页面整页状态

当前系统已经不适合继续用“项目 -> 仿真类型 -> 单 init-config”这种旧思路。

真正稳定的边界应是：

- session：基础会话
- 懒加载用户上下文：项目列表、资源池
- 项目上下文：阶段、参与人、项目文件信息
- 默认工况展开：默认参数链路
- 申请单实例状态：最终提交内容

## 5. 当前建议

后续配置系统演进时，应坚持以下原则：

1. 配置只负责默认值和可选范围
2. 快照只负责记录用户最终提交结果
3. 用户权限上下文与项目上下文严格分离
4. 旧接口只保留兼容包装，不再继续长逻辑

这样才能保证：

- 配置变更不会污染历史订单
- 快速提单链路稳定
- 结果页和编辑页能基于快照稳定回放
