# 配置驱动提单系统设计

面向当前项目“配置化、参数化、交互友好、提升提单效率”的主线维护。
**最后更新**: 2026-03-21

## 1. 设计目标

本系统的配置设计不是单纯做配置中心 CRUD，而是服务于以下目标：

1. 提单页面根据配置动态渲染
2. 用户可以在预设基础上高效完成参数化设置
3. 提交后的订单能够保留完整快照，支持编辑、查看和结果回读
4. 后续可以继续扩展自动化和人性化交互，而不必反复改底层数据模型

## 2. 核心原则

### 2.1 配置先于页面

页面不应写死为固定表单，而应由配置、关系和默认值解释出页面结构。

### 2.2 默认值优先

目标不是让用户从零开始填写，而是：

- 先给出项目默认上下文
- 再给出工况配置下的默认参数组
- 再给出默认输出组
- 再给出默认求解器
- 最后允许用户只改必要项

### 2.3 配置与快照分离

配置用于生成提单初始状态。
快照用于记录用户最终提交的真实内容。

这两者必须分离，否则配置变更会污染历史订单。

## 3. 当前配置分层

### 3.1 基础配置层

- 项目
- 仿真类型
- 参数定义
- 工况定义
- 输出定义
- 求解器
- 姿态类型
- 关注器件
- 资源池

### 3.2 组合配置层

- 参数组 `param_groups`
- 输出组 / 工况输出组
- 求解器候选关系

### 3.3 关系配置层

- 项目与仿真类型关系
- 仿真类型与参数组关系
- 仿真类型与输出组关系
- 仿真类型与求解器关系
- 姿态与仿真类型关系
- 姿态 + 仿真类型对应的工况配置

## 4. 当前真实提单主结构

当前提单页真实主结构更接近：

```text
项目
├─ 原始文件
├─ 原始姿态
├─ 目标姿态
│  └─ 仿真类型
│     └─ conditionId
│        ├─ params
│        ├─ output
│        ├─ solver
│        └─ careDeviceIds
└─ 项目级补充信息
```

因此，当前真正需要驱动页面的不是单一 `simTypeId`，而是：

- 目标姿态
- 仿真类型
- `conditionId`
- `conditionId` 下的完整配置节点

## 5. 配置如何驱动提单

### 5.1 当前实现

当前前端提单页并不是直接消费单一 `init-config` 接口，而是结合多组配置数据在前端拼装运行态。

这说明当前系统已经进入“前端解释层较重”的阶段。

### 5.2 上线前应先收口的边界

如果目标是最快上线，建议先收口以下边界：

- `inputJson` 是唯一主快照
- `conditionId` 是运行时主键
- `optParam` 只是兼容字段
- `conditionSummary` 只是列表摘要
- 编辑、详情、结果优先统一消费 `inputJson`

### 5.3 为什么不宜继续放大前端解释层

当前前端已经承担：

- 多数据源取数
- 默认值解析
- 工况组合解释
- 快照序列化
- 编辑回填

如果继续叠加新规则而不先收口协议，维护成本会快速上升。

## 6. 登录、SSO 与权限

### 6.1 当前定位

登录、SSO 和权限控制会直接影响系统是否可上线，但不属于提单配置化主链路本身。

它们的定位应是：

- 与提单配置化并行推进的 P0 基础能力
- 上线前必须收口的认证与权限边界
- 后续细粒度权限拆分的基础层

### 6.2 最小上线建议

- 统一登录入口
- 统一用户模型
- 统一权限模型
- 先支持管理员 / 普通用户两级角色

## 7. 最快上线建议

### 7.1 适合保留

- 当前 React 19 + TypeScript + Query + Zustand 体系
- 当前 Flask + SQLAlchemy + JWT 体系
- 当前 `orders` 表上的过渡字段设计
- 当前前端提单页树状运行态

### 7.2 不建议为上线先做的事情

- 前后端框架迁移
- 大规模配置中心重构
- 先做复杂权限系统再做主链路

### 7.3 应优先完成

1. 收口 `conditionId`
2. 收口 `inputJson`
3. 统一编辑页、详情页、结果页的快照消费
4. 登录 / SSO 收口
5. 再做配置解释层与页面层拆分

## 8. 后续演进方向

### P0

- 冻结提单主协议
- 冻结主快照结构
- 冻结登录 / 权限最小模型

### P1

- 抽离前端配置解释层
- 降低页面内硬编码逻辑
- 明确结果页迁移清单

### P2

- 推进更细粒度权限
- 推进自动化推荐与人性化交互
- 再评估是否需要升级初始化协议
