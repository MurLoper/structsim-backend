# 后端现状与开发计划

面向当前后端仓库状态维护。
**最后更新**: 2026-03-21

## 1. 当前现状

### 1.1 已有基础能力

- Flask 应用工厂和蓝图结构已成型
- `auth`、`config`、`orders`、`results`、`rbac`、`upload` 已按模块拆分
- JWT 认证链路已存在
- 登录模式接口、普通登录、SSO 登录交换接口已存在
- 订单列表、详情、创建、更新、统计、结果查询链路已存在
- 配置中心基础数据、组合关系和工况配置能力已存在

### 1.2 当前核心问题

- 普通登录与 SSO 登录仍是两套并行认证逻辑，尚未完全统一
- 提单页真实运行态已经是树状结构，但部分文档与部分接口仍按单 `simTypeId` 或平铺结构描述
- 订单创建已同时保留 `opt_param`、`input_json`、`condition_summary`，但主次边界还需要继续固化
- 结果链路虽已可用，但旧字段和旧消费方式仍残留，存在误用风险
- 数据库迁移和初始化流程仍偏依赖经验，不够标准化

## 2. 当前最重要的后端目标

当前后端优先级应聚焦于：

1. 登录与 SSO 的统一接入策略
2. 提单主链路协议收口
3. 订单快照结构稳定化
4. 结果链路统一消费入口
5. 上线前迁移与回归流程标准化

## 3. 登录与 SSO

### 3.1 当前判断

当前系统已经不是“是否需要 SSO”的问题，而是“如何把普通登录和 SSO 收口成统一认证边界”的问题。

### 3.2 当前建议

- 保留普通登录能力
- 保留当前 SSO 回调换票能力
- 统一登录模式配置输出
- 统一用户模型
- 统一权限模型

### 3.3 最小上线方案

- 一个登录入口
- 两种登录方式
- 一套用户态
- 两级角色：管理员 / 普通用户

## 4. 提单主链路

### 4.1 当前判断

真正影响上线的不是配置表多，而是提单主链路依赖关系是否稳定。

当前真实主链路更接近：

```text
项目 -> 目标姿态 -> 仿真类型 -> conditionId -> 配置节点
```

### 4.2 需要收口的接口

- `/api/v1/orders/init-config`
- `/api/v1/orders`
- `/api/v1/orders/:id`
- `/api/v1/results/*`

补充判断：

- `/api/v1/orders/init-config` 当前仍偏单 `simTypeId` 视角
- 当前前端提单主实现并不直接依赖该接口完成整页装配
- 上线前更重要的是先统一 `input_json`、`conditionId` 与结果消费边界

### 4.3 快照建议

订单创建时应明确区分：

- 主快照 `input_json`
- 兼容字段 `opt_param`
- 摘要字段 `condition_summary`
- 客户端元信息 `client_meta`

## 5. 后续开发计划

### P0

- 输出统一登录模式配置
- 对齐普通登录和 SSO 登录的用户模型
- 固化提单主协议
- 固化订单快照结构，明确 `input_json` 为主、`opt_param` 为兼容、`condition_summary` 为摘要
- 统一结果查询入口，明确旧接口废弃策略

### P1

- 初始化数据和迁移流程文档化
- 关键接口回归测试补齐
- 结果页与详情页快照消费统一

### P2

- 更细的权限与审计能力
- 结果链路进一步结构化
- 历史库查询边界进一步隔离

## 6. 评审建议

如果目标是尽快形成可上线版本，后端当前不建议进行框架迁移。

优先顺序应是：

1. 收口协议
2. 统一登录
3. 稳定提单
4. 固化结果消费
5. 再做迁移和重构
