# 后端现状与推进计划

最后更新：2026-04-10

## 1. 当前已收口的部分

### 1.1 登录与会话

- 登录与 SSO 已切到“拿 token 后统一拉 session”
- `auth/session` 已成为前端受保护页面的会话真源
- session 已带提单限制与最近常用项目

### 1.2 提单初始化

当前提单初始化已经拆成：

1. 用户基础会话
2. 提单页懒加载用户上下文
3. 项目上下文
4. 默认工况展开后的提单模板状态

其中项目上下文正式接口为：

```http
GET /api/v1/orders/init-project-config
```

旧的 `/api/v1/orders/init-config` 仅保留兼容包装。

### 1.3 资源池

- 资源池已从项目上下文中剥离
- 资源池改为提单页懒加载
- 未配置真实接口时，后端直接返回 mock 数据

## 2. 当前明确的业务边界

### 2.1 项目

- 项目不直接关联仿真类型
- 项目不直接关联 `conditionId`
- 项目主要负责：
  - 阶段
  - 参与人候选
  - 文件上下文

### 2.2 默认快速提单

默认快速提单继续由默认工况配置展开：

- 默认参数组
- 默认输出组
- 默认求解器

参数组和输出组的项目过滤规则是：

- 显式关联当前项目：生效
- 项目为空：视为全部项目可用

### 2.3 资源池与项目列表

这两类都属于用户权限上下文：

- 可见项目列表
- 资源池列表
- 默认资源池

不应放进首屏 session，也不应放进项目上下文。

## 3. 当前剩余重点

### P0

- 保持 `auth/session`、提单懒加载查询、项目上下文接口的职责边界稳定
- 避免任何新逻辑重新回流到旧的 `/orders/init-config`
- 继续收口文档和 OpenAPI，全部以新接口与新分层为准

### P1

- 将“可见项目列表”进一步语义化为显式的用户权限项目查询接口
- 为项目历史常选参与人补充更稳定的统计来源
- 补更多自动化回归测试

### P2

- 若未来公司接口就绪，再把本地项目列表替换为真实项目权限接口
- 若需要更强的快速提单能力，再评估是否引入独立的提单模板体系

## 4. 当前建议

后端后续继续按以下顺序推进：

1. 先保证提单初始化四层边界稳定
2. 再补权限接口与真实项目接口的替换能力
3. 最后再考虑独立模板体系和更复杂的策略推荐

不建议再把“项目上下文、用户权限上下文、申请单实例状态”重新揉回同一接口。
